0.1 范围（第一阶段将完成的工作）
- 支持在固定的LLM池（多个供应商，不同优势）中进行单轮路由，选择：
  - 一个主要（Top-1）模型，以及
  - 一个备用模型。
- 表示：
  - Q空间（Q-space） = 查询需求
  - M空间（M-space） = 模型能力
    并通过两个小型学习映射将两者投影到一个共享的Z空间（Z-space），然后使用向量运算在Z空间中进行相似度匹配。
    
- 优先优化答案质量，然后将成本/延迟/安全性作为约束条件和次要权重。
- 保持架构的可扩展性，以支持未来的策略（级联、集成、多臂老虎机），但不在第一阶段实现它们。
0.2 范围之外（第一阶段不会做的工作）
- 不包含在线强化学习/老虎机算法
  
    仅进行离线训练 + 定期重训练。
    
- 不包含复杂的专家混合编码器或定制的大型模型
  
    使用现有的文本嵌入 + 小型投影层。
    
- 不包含超细粒度的标签分类法
  
    任务类型和领域是故意设计得比较粗略的。
    
- 不包含为每个租户定制的路由模型
  
    租户偏好被编码为特征，而不是独立的模型。

1. 架构快照
用一句话概括第一阶段架构：
查询 → Q空间编码器（嵌入 + 学习投影 f_Q）→ z_Q

模型 → M空间分析器（探测分数 + 学习映射 f_M）→ z_M

在共享的Z空间中匹配（点积）→ 应用成本/延迟/安全约束 → 选择Top-1 + 备用模型

1.1 核心组件
1) Q空间编码器
一个小型学习型编码器，用于将查询映射到Z空间。
- 输入
  - 原始查询文本
  - 租户信息（调用方是谁，其偏好）
- 输出
  - z_Q：用于在Z空间中进行相似度匹配的d维潜在向量
    
      （由文本嵌入之上的投影模型 f_Q 生成）
      
  - q_meta：可解释的特征，例如：
    - 任务类型
    - 包含代码标志
    - 长度分桶
    - 粗略领域
    - 安全敏感度
    - 租户偏好（成本 vs 延迟 vs 质量）
2) M空间注册表
对于每个模型，我们存储一条记录，该记录通过一个从探测结果到Z空间的小型映射生成。
- z_M：d维潜在能力向量
  
    （由映射模型 f_M 根据探测集性能和能力特征计算得出）
    
- m_meta：可解释的标量属性，例如：
  - 每个Token的成本
  - 典型延迟
  - 安全评级
  - 最大上下文长度
  - 租户可用性（哪些租户可以使用此模型）
3) 评分与路由
- 在Z空间中计算 quality_score：
  
    quality_score(q, m) = dot(z_Q, z_M)
    
- 使用 m_meta 应用：
  - 硬过滤器：安全性和上下文长度
  - 软惩罚：成本和延迟
- 为每个模型生成 final_score(q, m)。
- 选择：
  - Top-1模型作为主要模型
  - 次优模型作为备用模型
4) 训练与数据
- 使用公共基准测试 + 合成数据 + LLM裁判比较来构建静态训练集。
- 离线训练两个投影映射 f_Q (Q → Z) 和 f_M (M → Z)，使用对比/排序目标；评分固定为Z空间中的点积。
- 上线后：
  - 记录查询、路由决策和输出。
  - 使用日志 + LLM裁判生成更多标签。
  - 定期重训练 f_Q / f_M。

2. Q空间（查询需求）
2.1 Q空间编码的内容
Q空间回答以下问题：
“这个特定的查询需要模型具备哪些能力？”

在第一阶段，我们关注沿以下维度的粗粒度信息：
- 任务类型
  
    示例：聊天、代码、数学、翻译、工具类（期望如JSON等结构化输出）等。
    
- 推理深度
  
    低 / 中 / 高 – 大致需要多少步多步推理。
    
- 领域
  
    通用 / 编程 / 数学 / 金融 / 法律 / 医疗 / 其他。
    
- 安全敏感度
  
    普通 / 敏感 / 高风险（例如，自残、暴力）。
    
- 代码标志
  
    查询是否包含代码或明确要求代码。
    
- 长度分桶
  
    短 / 中 / 长 – 对上下文长度需求的粗略估计。
    
- 租户偏好
  
    该租户是优先考虑成本、延迟还是质量。
    
2.2 表示方法（混合：可解释 + 潜在）
第一阶段采用由学习型编码器 f_Q 实现的混合设计：
可解释模块（约 16–32 维）
- 编码：
  - 任务类型
  - 领域
  - 推理级别
  - 安全级别
  - 包含代码标志
  - 长度分桶
  - 租户偏好
    
    使用独热编码或分桶特征。
    
- 用于：
  - 过滤
  - 调试
  - 硬编码规则（例如，“如果敏感，则强制使用更安全的模型”）。
潜在模块（约 64 / 128 维）
- 使用文本嵌入模型，如 OpenAI text-embedding-3-small / ada（或同等模型）来获取高维基础嵌入。
- 将原始嵌入通过一个小型可训练投影层（例如，单个MLP）——此投影层是 f_Q 的一部分——以获得大小为 d_latent ≈ 64 / 128 的潜在向量。
- 捕捉未被明确建模为离散特征的细粒度语义和能力相关信号。
第一阶段设计选择（Q空间）
- latent_part = embedding → 可训练投影（64 / 128 维）
- interpretable_part ≈ 16–32 维
- Q_vector = concat(interpretable_part, latent_part)
  
    （在Z空间相似度中使用的是由 f_Q 产生的潜在模块）
    
2.3 Q空间编码器流程（第一阶段）
f_Q 的逻辑流程：
1. 预处理查询
  - 基本文本清理
  - 长度统计
  - 简单的安全/敏感度检测
2. 生成可解释特征（规则或轻量级分类器）
  - 任务类型（关键词 / 小型分类器）
  - 包含代码标志（检测代码模式 / 关键词）
  - 长度分桶
  - 安全标志
  - 租户偏好（来自租户配置）
3. 调用嵌入服务
  - 使用ada风格的嵌入（或同等模型）获取高维向量。
4. 应用投影层
  - 通过学习到的投影将嵌入映射到64 / 128维的潜在向量 q_latent。
5. 拼接成最终的Q向量
  - Q_vector = [q_interpretable ; q_latent]
    
      并取 z_Q = q_latent 作为查询在Z空间中的位置。

3. M空间（模型能力）
3.1 M空间编码的内容
M空间回答以下问题：
“该模型在我们关心的能力维度上表现如何？”

在第一阶段，我们维护一个静态能力画像（无持续的在线更新）。
- 潜在能力向量 z_M（约 64 / 128 维）
  - 根据模型在小型探测集上的表现学习得出，该探测集涵盖：
    - 数学
    - 代码
    - 通用问答
    - 重推理任务
    - 安全敏感提示
  - 由一个小型学习映射 f_M 生成，该映射将探测集分数/能力特征转换到Z空间，并使用对比/排序目标进行训练，使得：
    - 能力相似的模型在潜在空间中距离相近。
- 可解释的标量（在潜在向量之外）
  - 每1k token的成本
  - 平均延迟（例如，P50，可选P95）
  - 安全评级（来自内部评估）
  - 最大上下文长度
  - 租户可用性（哪些租户可以使用此模型）
3.2 静态表示（第一阶段）
第一阶段固定以下设计：
- z_M 的维度 = Q潜在模块的维度（例如，均为64 / 128）。
- z_M 通过探测集任务分数经由固定映射函数或小型网络 f_M 推导得出，该函数/网络离线训练一次后供所有模型复用。
- 成本 / 延迟 / 安全 / 上下文限制保持为独立的标量字段，不混入潜在向量。
- M空间在第一阶段是完全静态的：
  - 当模型升级时：
    - 重新运行探测集以获取更新的分数。
    - 使用相同的 f_M 重新计算 z_M。
    - 更新注册表，无需重训练整个路由器。
3.3 新模型接入（冷启动）
当添加新模型时：
1. 从元数据和模型家族初始化
  - 如果它是现有模型的升级版本：
    - 从旧模型的M记录开始作为基线。
    - 根据供应商信息（新的上下文长度、价格、基准测试结果）调整字段。
2. 运行预定义的探测集
  - 探测集大小：150–300个查询，涵盖主要能力维度：
    - 数学 / 代码 / 通用问答 / 重推理 / 安全敏感
  - 对于每个探测查询：
    - 运行新模型。
    - 使用以下方式将输出与参考模型或基本事实进行比较：
      - LLM裁判，或
      - 存在基本事实时的直接评估。
    - 汇总每个任务的分数或胜率。
  - 将这些分数输入预训练的映射 f_M 以生成 z_M。
探测集完成并计算出 z_M 后，模型即可添加到路由池中，无需重训练整个路由器。我们仅使用现有的 f_M 重新计算其M空间记录。

4. 共享Z空间与路由逻辑
4.1 共享潜在空间Z
我们定义一个维度为 d 的共享潜在空间 Z：
- Q编码器 f_Q 输出 z_Q ∈ ℝ^d
- M分析器 f_M 输出 z_M ∈ ℝ^d
- 能力匹配通过此空间中的点积相似度完成
Z空间本身不是另一个模型；它是 z_Q 和 z_M 被 f_Q 和 f_M 投影后所处的公共坐标系。于是，路由变成了一个“为此查询向量找到最兼容的模型向量”的问题。
4.2 评分
质量分数
对于每个（查询，模型）对：
- quality_score(q, m) = dot(z_Q, z_M)
这衡量了模型能力与查询需求的匹配程度。在第一阶段没有额外的神经评分模型；我们依赖Z空间中的点积。
最终路由分数
我们将质量与成本和延迟结合起来：
- final_score(q, m) = quality_score(q, m) - α * cost(m) - β * latency(m)
其中：
- 安全性和上下文长度作为硬过滤器应用：
  - 如果 safety_rating(m) 低于阈值且查询为高风险 → 不考虑该模型。
  - 如果 max_context(m) 低于所需上下文长度 → 不考虑该模型。
4.3 路由策略（第一阶段）
- 对于所有通过安全和上下文过滤器的模型：
  - 计算 final_score(q, m)。
- 选择：
  - primary_model = 最终分数最高的模型
  - fallback_model = 最终分数第二高的模型
- 执行主要模型。如果出现以下情况，自动切换到备用模型（并记录事件）：
  - 超时
  - API错误
  - 检测到严重安全问题
- 记录所有路由决策，包括：
  - 查询
  - 选择的模型和备用模型
  - 输出
  - 用户反馈（例如，点赞/点踩）
  - 错误
第一阶段为路由固定以下选择：
- 使用点积作为核心相似度度量。
- 仅在计算潜在相似度后应用成本和延迟；它们不参与嵌入训练。
- 仅使用 Top-1 + 备用模型进行路由：
  - 不进行并行多模型调用。
  - 不使用集成方法。

5. 数据与训练计划（第一阶段）
5.1 数据来源（静态）
第一阶段需要一个具有合理覆盖范围的中等规模静态数据集来启动路由器。
1) 公共基准测试
- 示例：
  - MMLU
  - BIG-Bench
  - 代码、数学、翻译数据集
- 用于：
  - 标注粗略的领域/技能类别（例如，数学、法律、代码）
  - 为哪些查询类型偏好哪个模型家族提供冷启动信号
2) LLM裁判比较
- 对于同一查询：
  - 运行多个模型。
  - 请一个强大的LLM来比较答案。
- 产生：
  - 成对偏好，例如“模型A优于模型B”
  - 该查询的首选模型标签
- 这是持续扩展训练数据集的主要方式。
3) 合成查询
- 例如：
  - “生成20个需要大量算术推理的查询”
  - “生成20个关于美国法律的查询”
- 这些查询带有已知的能力标签，可用于：
  - 在稀有但重要的技能上训练Q空间（包括可解释和潜在部分）。
5.2 训练方法（第一阶段）
第一阶段专注于两种训练方法，都针对两个投影映射 f_Q 和 f_M。
1) 对比/排序（核心）
- 对于每个查询：
  - 根据评估结果确定一个或多个“更好”的模型和“更差”的模型。
  - 训练 f_Q 和 f_M 的参数，使得：
    - (z_Q, z_M_better) 在Z空间中更接近。
    - (z_Q, z_M_worse) 在Z空间中更远离。
- 这与点积评分直接对齐，并优化了候选模型的排序质量。
2) 轻量级分类（可选）
- 使用 z_Q 通过softmax分类器预测“哪个模型是最好的”（在z_Q之上添加辅助头）。
- 或者使用 z_M 预测模型在哪些能力分桶上表现强劲，以稳定M空间并检测不一致性。
第一阶段在训练上保持纯离线：
- 不使用强化学习或基于老虎机的在线学习。
- 主要标签来源是LLM裁判输出和基准测试结果。
- 人工标注保留用于抽样和质量检查，而非批量标注。
